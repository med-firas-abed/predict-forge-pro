────────────────────────────────────────────────────────────────────────────────
   OVERVIEW: WHAT IS THIS PIPELINE?
────────────────────────────────────────────────────────────────────────────────

Goal: predict when an industrial elevator will fail BEFORE it actually breaks.

Simple analogy: Imagine your car dashboard — but instead of just a red light
that turns on when the engine is already overheating, we have an intelligent
system that says "your engine will overheat in 12 days" and grades its current
health from 0% to 100%, every second, based on vibration, power consumption,
temperature, and humidity readings.

The pipeline has 7 steps:
  Step 1: Simulate realistic sensor data (because we don't have years of real failure data)
  Step 2: Transform raw sensor readings into meaningful features
  Step 3: Detect anomalies using Isolation Forest + a hybrid ensemble
  Step 4: Convert anomaly scores into a human-readable Health Index (0–100%)
  Step 5: Predict Remaining Useful Life (RUL) — "how many days until failure?"
  Step 6: Evaluate everything with metrics, plots, and SHAP explainability
  Step 6B: Validate the pipeline on NASA's public CMAPSS benchmark
  Step 7: Package everything into a production-ready engine for FastAPI

    3. Eventually: short circuits, bearing wear, motor failure
  This is why the REGRESSION is driven by current level.

                               IF gets 60% weight because it considers ALL 12

                               the most direct physical indicator of bearing health.
                               Why not 50/50? IF catches subtle multi-sensor patterns
                               that RMS alone misses, so it deserves more weight.

HEALTH INDEX THRESHOLDS:
  HI_EXCELLENT     = 0.8      ≥ 80% → green zone, no action needed.
  HI_GOOD          = 0.6      ≥ 60% → yellow, schedule maintenance soon.
  HI_CRITICAL      = 0.3      ≥ 30% → orange/red, urgent attention required.
  HI_SMOOTH_WINDOW = 120      2-minute rolling average (120 samples at 1 Hz).

  Why these thresholds?
  - 0.8 (80%): ISO 10816 vibration standard considers machines with < 1.8 mm/s
    RMS as "good". Our simulation maps HI ≥ 0.8 to RMS ∈ [0.8, 1.5] mm/s.
  - 0.6 (60%): transition zone where degradation is detectable but not critical.
    Mapped to RMS ∈ [1.5, 2.0] mm/s — scheduled maintenance is recommended.
  - 0.3 (30%): machines below this are in dangerous condition.
    Mapped to RMS > 4.5 mm/s — risk of sudden failure.
  - 120s smoothing window: removes high-frequency noise with minimal lag.
    A 2-minute average covers ~2.7 cycles (44s each), enough to smooth
    within-cycle variance without introducing excessive phase delay.
    Previously 600s (10 min) but this caused too much lag and reduced
    HI-to-ground-truth correlation.

LEVEL 1 — SIMULATION (Step 1): Motor specs → realistic synthetic data

  The motor's physical characteristics directly shape every simulated signal:

  • POWER CURVES come from the motor + LOAD:
    - At empty: P = 0.30 kW (friction/inertia only)
    - At full load (285 kg) healthy: P = 1.51 kW
    - At full load degraded: P = 2.16 kW (worn bearings increase friction → I²R losses)
    - P_descent = 0.35 kW (gravity assists, motor barely works)
    - P_pause = 0 kW (motor off)
    - Degraded maximum = 2.16 kW (near rated current of 3.90 A)
    → These come from the plaque signalétique (400V Y-winding, 1410 RPM, cosφ = 0.80)
      and the load equation: P = P_empty + P_load_range × (charge/285) + P_deg × (1 - HI)
    - CURRENT derived: I = P / (√3 × 400 × 0.80) = P / 554.3
      This is output as current_a in the simulation for direct analysis.

  • VIBRATION RANGES come from ISO 10816 + the motor's shaft frequency:
    - Shaft frequency = (1410 RPM / 60) / 60 = 0.392 Hz
    - Healthy RMS: 0.8 – 1.5 mm/s (ISO 10816 Class I for motors < 15 kW)
    - Critical RMS: 4.5 – 7.5 mm/s (ISO 10816 Zone D = danger)
    - Chassis resonance at 3 Hz adds ±0.2 mm/s periodic component
    → The vibration bands are motor-class-specific, not generic.

  • CYCLE TIMING comes from factory observation:
    - 12s ascent, 12s descent, 20s pause = 44s total cycle
    - This 44s rhythm creates the characteristic power waveform:
      [HIGH – LOW – ZERO – HIGH – LOW – ZERO – ...]
    → The simulation reproduces this exact pattern at 1 Hz resolution.

  • THERMAL MODEL comes from the motor + local climate:
    - 3.5°C motor surface rise per kW (proportional to power dissipation)
    - Ambient: 14–28°C (Ben Arous, Tunisia, spring)
    - Humidity: 55–80% RH (Mediterranean coastal)
    → A degraded motor runs hotter BECAUSE it draws more power.

  Result: 100 simulated machine lifetimes that behave like the REAL SITI motor
  — not like an abstract academic exercise.

LEVEL 2 — PREPROCESSING (Step 2): Motor baseline calibrates the scaler

  After extracting 12 features from the raw signals, we normalize them with
  z-score: X_norm = (X - μ_healthy) / σ_healthy.

  Where do μ and σ come from? From the FIRST HOUR of operation where the
  simulated motor is known to be healthy (HI ≥ 0.8). Because the simulation
  was calibrated to the real motor (Level 1), these μ/σ values represent:

  • μ_rms ≈ 1.1 mm/s  → the healthy vibration of THIS motor class (ISO 10816)
  • μ_power ≈ 1.51 kW → the nominal consumption of THIS motor
  • μ_temp ≈ 22°C     → average motor surface temperature in THIS climate
  • σ values represent the normal variation for each sensor under healthy
    operating conditions — tight for power (±0.5%), wider for vibration (±1.5%)

  These are stored in models/scaler_params.json and used FOREVER after:
  in production, every incoming sensor reading is normalized against the
  SITI FC100L1-4's healthy baseline. A reading that would be "fine" for a
  10 kW motor might be "alarming" for our 2.2 kW motor — the scaler knows
  the difference because it was trained on motor-specific simulation.

  Key insight: The scaler is NOT generic. It encodes the physical signature
  of this specific motor class. A different motor would need re-simulation
  (with its own parameters in config.py) to produce a new scaler.

══════════════════════════════

LEVEL 3 — PRODUCTION ENGINE: Motor parameters protect against bad data

  In real-time operation, the prediteq_engine.py class uses the motor's
  physical limits for quality control:

  • OUTLIER REJECTION (5σ rule):
    If a sensor reading is more than 5 standard deviations from the healthy
    mean (μ from scaler_params.json), it is REJECTED as a sensor glitch:
      → rms = 999 mm/s? Impossible for a 2.2 kW motor. Rejected.
      → power = -3 kW? Impossible. Rejected.
    The 5σ threshold is calibrated to THIS motor's normal range, not a
    universal constant. For our SITI motor, 5σ_rms ≈ 5 mm/s, so any reading
    above ~6 mm/s from a healthy baseline is flagged as a sensor error.

  • HI NORMALIZATION BOUNDS (p5/p95):
    The HI formula uses percentile bounds from training data. These bounds
    reflect what "fully healthy" and "severely degraded" look like FOR THIS
    MOTOR. A different motor class would have different bounds.

  • RUL CONFIDENCE INTERVALS:
    The Random Forest's 200 trees were trained on degradation patterns that
    follow this motor's power curves, vibration ranges, and thermal behavior.
    The confidence intervals (10th–90th percentile of tree predictions) are
    calibrated to the uncertainty of THIS motor type.

SUMMARY: ONE NUMBER'S JOURNEY (example: MOTOR_POWER_KW = 2.2)

  1. Datasheet says: SITI FC100L1-4 = 2.2 kW rated power
  2. config.py defines: MOTOR_POWER_KW = 2.2
  3. Step 1 uses it to compute:
     - P_ascent_nominal = 1.51 kW
     - P_ascent_degraded = 2.16 kW (+ friction losses)
     - P_descent = 0.35 kW
  4. Step 2 observes: μ_power ≈ 1.51 kW over healthy trajectories
     → stores this in scaler_params.json

     is rejected as a sensor error

  So the single number "2.2 kW" from the motor nameplate ultimately determines:
  - What "normal" power consumption looks like
  - When the HI starts declining
  - When anomaly alerts fire
  - When we predict the motor will fail

  The same tracing applies to every physical parameter in config.py:
  RPM, load, cycle timing, reducer ratio, power factor — they ALL flow from
  the datasheet through simulation, into the scaler, and finally protect the
  production engine.


════════════════════════════════════════════════════════════════════════════════
   STEP 1 — DATA SIMULATION (step1_simulate.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  We don't have years of real elevator failure data — no factory lets their
  machines break on purpose for data collection. Instead, we simulate 100
  "virtual machines", each going from brand-new (100% health) to broken (0%).
  We use physics-based models so the data behaves like real sensors would.

Technical:

  WHY SIMULATE?
  In predictive maintenance, the biggest challenge is data. Real failure data
  requires machines to actually fail, which is expensive and rare. Simulation
  (also called digital twins or synthetic data) is the industry-standard
  approach for training initial models. The key is ensuring the simulation is
  physically grounded, not random.

  FOUR DEGRADATION PROFILES:
  Real machines don't all break the same way, so we model 4 patterns:

  A — Linear:       HI drops steadily over time. Represents gradual wear of
                     bearings, belts, or gears. Like a tire slowly losing tread.
                     Formula: HI = 1 - 0.7 × (t/t_fail)

  B — Exponential:  HI stays high for a long time, then drops rapidly near the
                     end. Represents fatigue failure — cracks propagate slowly
                     then suddenly everything breaks. Common in metal components.
                     Formula: HI = 1 - 0.7 × (t/t_fail)²

  C — Stepwise:     HI drops in sudden jumps (5 discrete steps). Represents
                     shock events — e.g. overloads, power surges, or impacts.
                     Each event permanently degrades the machine.
                     Formula: HI = 1 - 0.85 × floor(5 × t/t_fail) / 5
                     Minimum HI = 0.15 (definitively crosses Critical threshold)

  D — Noisy linear: Same as A, but with Gaussian noise (σ=0.08) added.
                     Represents real-world measurement uncertainty where the
                     true degradation trend is hard to see through noise.
                     Formula: HI = (1 - 0.7 × t/t_fail) + N(0, 0.08)

  Why 4 profiles and why these? Because the ML model must generalize — if we
  only trained on linear degradation, it would fail to predict exponential or
  stepwise failures. These 4 cover the most common patterns in industrial
  reliability literature (Weibull, Paris' Law, discrete-event).

  SENSOR GENERATION FROM HI:

  1. Vibration (RMS, mm/s): Mapped directly from HI:
     - HI ≥ 0.8 → RMS ∈ [0.8, 1.5] mm/s (healthy, per ISO 10816)
     - HI ≥ 0.6 → RMS ∈ [1.5, 2.0] mm/s (early wear)
     - HI ≥ 0.3 → RMS ∈ [2.0, 4.5] mm/s (degraded)
     - HI < 0.3 → RMS ∈ [4.5, 7.5] mm/s (critical/failure)
     Plus chassis resonance (±0.2 mm/s at 3 Hz) and sensor noise (±1.5%).

  2. Power (kW): During ascent, a degraded motor works harder:
     P_degraded = P_nominal + 0.65 × (1 - HI)
     At HI = 1.0: P = 1.51 kW (nominal)
     At HI = 0.0: P = 1.51 + 0.65 = 2.16 kW (degraded maximum)
     This mirrors the physics: worn bearings → more friction → more current.
     During descent (0.35 kW) and pause (0 kW), power is constant regardless
     of health — degradation only shows during load-bearing ascent.

  3. Temperature (°C): Ambient daily cycle + motor heat dissipation:
     T = T_ambient(t) + 3.5 × (P_current / P_max_degraded)
     The 3.5°C factor models motor surface heating proportional to power draw.
     A degraded motor runs hotter because it draws more power (I²R losses).

  4. Humidity (%RH): Follows an inverse daily cycle (highest early morning,
     lowest afternoon — standard Mediterranean pattern).

  Each trajectory has a random failure time that is capped at 0.95 × t_max
  to guarantee ALL trajectories reach failure (HI < 0.3). The trajectory then
  runs to min(t_fail × 1.10, t_max) to capture the Critical zone. A random
  factor U(0.75, 0.95) is applied to vary the failure time across trajectories.
  This ensures load diversity doesn’t prevent light-load trajectories from
  reaching failure within the trajectory window.

  OUTPUT: data/raw/trajectories.csv — 100 trajectories × ~800 min × 1 Hz.
  Columns: trajectory_id, profile, t_seconds, rms_mms, power_kw, temp_c,
           humidity_rh, simulated_hi, phase.


════════════════════════════════════════════════════════════════════════════════
   STEP 2 — PREPROCESSING & FEATURE ENGINEERING (step2_preprocess.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  Raw sensor numbers (like "1.3 mm/s vibration") are not very informative by
  themselves. We extract 12 "features" that capture trends, variability, energy
  consumption, and correlations between sensors — things that reveal the
  machine's true condition.

  Analogy: When a doctor examines you, they don't just look at your current
  heart rate. They also check: is it increasing? how much does it vary? is it
  correlated with your blood pressure? That's what feature engineering does.

Technical — the 12 features:

  VIBRATION FEATURES (from VTV122 accelerometer):
  1. rms_mms          Direct RMS vibration value (mm/s).
                       Most fundamental indicator — directly measures how much
                       the machine is shaking.

  2. drms_dt           First time derivative of RMS — how FAST is vibration
                       changing? A sudden spike (large drms_dt) means something
                       just broke or shifted. Computed as: rms[t] - rms[t-1].

  3. rms_variability   Rolling 60-second standard deviation of RMS. High
                       variability = unstable vibration = possible looseness or
                       intermittent contact. Healthy machines have LOW variability.

  POWER FEATURES (from Siemens PAC2200 power meter):
  4. p_mean_kw         Rolling 60-second mean power. Tracks the average energy
                       consumption trend. Degraded motors draw more power.

  5. p_rms_kw          Rolling 60-second RMS of power. Captures both the mean
                       AND the variation of power — more informative than mean
                       alone because power surges (during ascent starts) contain
                       health information.

  6. dp_dt             First derivative of power. Detects sudden load changes
                       or motor efficiency drops.

  ENERGY & TIMING FEATURES:
  7. e_cycle_kwh       Energy consumed per ascent cycle (kWh), computed by
                       trapezoidal integration: ∫P(t)dt over each ascent phase.
                       WHY? A degraded motor uses more energy per cycle because
                       friction is higher. This is the most reliable degradation
                       metric according to reliability literature.

  8. duration_ratio    Actual ascent duration / nominal 12 seconds.
                       If this ratio > 1.0, the motor is taking LONGER than
                       designed to complete an ascent — it's struggling.
                       If < 1.0 or = 1.0, the motor is healthy.

  THERMAL FEATURES (from Vaisala HMT370EX):
  9. t_mean_c          Rolling 5-minute (300-second) mean temperature. Filters
                       out short-term fluctuations to reveal the true thermal
                       trend. Uses a wider window than vibration because
                       thermal changes are slower (thermal inertia).

  10. dt_dt            Temperature rate of change (°C/min). Rapid heating
                       indicates excessive friction or electrical losses.
                       Computed over a 60-second shift.

  HUMIDITY & CROSS-CORRELATION:
  11. hr_std           Rolling 60-second humidity standard deviation. High
                       humidity variability can indicate ventilation problems or
                       environmental control failure (which accelerates corrosion).

  12. corr_t_p         Pearson correlation between temperature and power over a
                       60-minute (3600-second) rolling window. In a healthy
                       machine, temp and power are moderately correlated (motor
                       heat follows power draw). If this correlation breaks
                       (very high or very low), it suggests abnormal thermal
                       behavior — e.g. blocked ventilation, insulation failure.

  Z-SCORE NORMALIZATION:
  After computing all 12 features, we normalize each one using z-score:
    X_norm = (X - μ_healthy) / σ_healthy

  μ and σ are computed ONLY from healthy data (first hour + HI ≥ 0.8).
  Why? Because we want "0" to mean "normal" and ">2" to mean "2 standard
  deviations above healthy". This makes the Isolation Forest's job easier and
  makes features comparable across different scales (mm/s vs kW vs °C).

  OUTPUT: data/processed/features.csv (100 trajectories × 12 features + 12 normalized)
          models/scaler_params.json (μ and σ for each of 12 features)


════════════════════════════════════════════════════════════════════════════════
   STEP 3 — ANOMALY DETECTION: ISOLATION FOREST + HYBRID ENSEMBLE
                                    (step3_isolation_forest.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  Now we need the AI to learn what "healthy" looks like, so it can recognize
  when something goes wrong. We use an algorithm called Isolation Forest that
  learns exclusively from healthy data — it never sees broken machines during
  training. Then when we show it new data, it tells us "this looks normal" or
  "this looks strange".

  But relying on just one method is risky, so we combine two detection methods
  (Isolation Forest + vibration threshold) into a "hybrid ensemble" — like
  getting a second opinion from another doctor.

Technical:

  WHY ISOLATION FOREST?
  - It's unsupervised: trained ONLY on normal data. In predictive maintenance,
    we have abundant healthy data but very few failure examples. Supervised
    methods (like neural networks) need balanced classes — we don't have that.
  - It's fast: O(n × log(n)) per tree. Important for real-time scoring.
  - It works on all 12 features simultaneously, catching multi-dimensional
    anomalies that no single feature threshold would detect.

  HOW IT WORKS (simplified):
  IF builds 100 random trees. Each tree randomly picks a feature and a split
  point to isolate data points. Normal points need MANY splits to isolate
  (they're surrounded by similar data). Anomalies need FEW splits (they're
  already "isolated" in feature space). The anomaly score is the average
  number of splits needed — fewer splits = more anomalous.

  TRAINING DATA SELECTION:
  We train ONLY on data where:
    - t_seconds ≤ 3600 (first hour of operation)
    - simulated_hi ≥ 0.8 (known healthy state)
  This ensures the model learns a clean "healthy baseline". Training on all
  data would contaminate the model with degraded patterns.

  TRAIN/TEST SPLIT:
  80 trajectories for training, 20 for testing. Split by TRAJECTORY, not by
  random rows — otherwise the model could "leak" knowledge from the same
  trajectory's future into its training set.

  THE HYBRID ENSEMBLE:
  IF alone has a weakness: it sometimes flags healthy ascent phases as anomalies
  (false positives) because the power spikes during ascent look "unusual" even
  though they're normal. We add a second detector:

  RMS z-score: (rms_current - μ_healthy) / σ_healthy
    - Simple, physics-based: high vibration = bad health. Period.
    - Very few false positives because vibration is the most direct indicator.
    - But it's one-dimensional — it misses subtle multi-sensor patterns.

  COMBINING THEM:
    hybrid_score = 0.6 × IF_normalized + 0.4 × RMS_z_normalized

  Both components are normalized to [0, 1] using robust percentiles (p1, p99)
  from the training set, so they contribute on the same scale.

  WHY 60/40?
  IF gets more weight because it considers all 12 features (including power
  patterns, temperature trends, and cross-correlations) — it catches anomalies
  that RMS alone would miss. But RMS provides a strong physical ground truth
  that anchors the score. We tested 50/50, 70/30, and 60/40 — the 60/40 split
  gave the best F1 score on the validation set.

  THRESHOLD OPTIMIZATION:
  We find the optimal hybrid_score threshold that maximizes F1 score on the
  training set. Ground truth: simulated_hi < 0.6 = degraded. The threshold
  search scans from 0.20 to 0.85 in 0.01 increments.

  EARLY DETECTION:
  We also measure "lead time" — how much earlier does IF/hybrid detect
  degradation compared to the simple RMS threshold? Typically 15–30 minutes
  earlier, which in a factory running 8h/day translates to hours of advance
  warning.

  RESULTS (on test set):
  - IF-only:     Precision ~0.77, but some false positives during healthy ascents
  - RMS-only:    Precision ~0.90, but detects late (only when vibration is severe)
  - Hybrid (AND): Very high precision (~0.93), but lower recall
  - Hybrid (weighted): Best F1 balance — Precision ~0.82, Recall ~0.85

  OUTPUT: data/processed/anomaly_scores.csv (all scores, flags, hybrid scores)
          models/isolation_forest.pkl (trained IF model)
          models/hybrid_params.json (alpha, threshold, normalization params)


════════════════════════════════════════════════════════════════════════════════
   STEP 4 — HEALTH INDEX (step4_health_index.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  Anomaly scores are abstract numbers. A factory manager doesn't want to hear
  "your anomaly score is 0.73." They want to hear "your machine is at 42%
  health — Degraded zone." This step converts the technical score into a simple
  0–100% Health Index with color-coded zones.

Technical:

  CONVERSION FORMULA:
    HI_raw = 1 - (hybrid_score - p5) / (p95 - p5)
    HI_raw = clip(HI_raw, 0, 1)

  Where p5 and p95 are the 5th and 95th percentiles of the hybrid_score on
  training data. This robust normalization:
  - Uses percentiles (not min/max) to be resistant to outliers
  - Maps the healthy distribution to HI ≈ 1.0
  - Maps severe anomalies to HI ≈ 0.0
  - Why p5/p95? Because p0/p100 would be sensitive to a single extreme value.
    5th/95th percentiles capture 90% of the training distribution, which is
    the standard "robust" choice in signal processing.

  SMOOTHING:
    HI_smooth = rolling_mean(HI_raw, window=120)
  
  A 120-sample (2-minute) rolling average. Why?
  - Raw HI fluctuates second-to-second due to sensor noise and cycle phases
    (HI dips slightly during ascent even on a healthy machine)
  - A 2-minute window covers ~2.7 complete cycles (44s each), enough to
    smooth within-cycle noise while minimizing lag
  - Previously 600s (10 min) but this caused excessive smoothing that blurred
    the degradation signal and reduced HI-to-truth correlation (0.82 → 0.855)
  - Adaptive at start: uses min_periods=1, so the first sample has a "window"
    of 1, growing to 120 — no waiting period before HI is available

  DOWNSAMPLING:
  After smoothing, we keep only 1 value per minute (every 60 seconds).
  Why? For RUL prediction, minute-level resolution is more than enough, and it
  makes the dataset 60× smaller (faster training, less memory).

  ZONE LABELING:
  - HI ≥ 0.80 → Excellent (green)
  - HI ≥ 0.60 → Good (yellow)
  - HI ≥ 0.30 → Degraded (orange)
  - HI < 0.30 → Critical (red)

  VALIDATION:
  Pearson correlation between HI_smooth and simulated_hi is ~0.855.
  This means our data-driven HI closely tracks the "true" health — the pipeline
  successfully recovers the ground truth from noisy sensor data alone.

  OUTPUT: data/processed/hi.csv (HI per trajectory, per minute, with zone labels)
          models/hi_params.json (p5 and p95 normalization parameters)


════════════════════════════════════════════════════════════════════════════════
   STEP 5 — RUL PREDICTION (step5_rul_model.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  Now we can answer the big question: "How many days until this machine fails?"
  We train a Random Forest model that looks at the current sensor readings AND
  the Health Index trend over the last hour, and predicts how many minutes of
  operating time remain before HI drops below 30% (Critical zone).

  The answer comes with a confidence interval: "142 days ± [120 — 165 days]"

Technical:

  WHY RANDOM FOREST?
  - It's an ensemble of 300 decision trees — each tree gives a prediction,
    and we average them. This reduces variance and overfitting.
  - It naturally provides uncertainty estimates: the spread between individual
    tree predictions gives us a confidence interval (no extra computation needed).
  - It handles mixed feature types and nonlinear relationships well.
  - It's interpretable with SHAP (Step 6).
  - We chose 300 trees (n_estimators=300), max_depth=12, and min_samples_leaf=10
    as a balance: enough complexity to capture real patterns, but regularized
    to prevent overfitting on 100 trajectories. Compared to GradientBoosting
    (R²=0.984) — RF won (R²=0.985).

  FEATURE VECTOR (17 features per prediction):
  12 normalized sensor features (from Step 2):
    rms_mms, drms_dt, rms_variability, p_mean_kw, p_rms_kw, dp_dt,
    e_cycle_kwh, duration_ratio, t_mean_c, dt_dt, hr_std, corr_t_p

  5 HI summary features (from 60-minute lookback window):
    - hi_now:   current HI value
    - hi_mean:  average HI over last 60 min
    - hi_std:   standard deviation of HI (how stable/volatile is health)
    - hi_min:   minimum HI seen in last 60 min (worst recent state)
    - hi_slope: linear regression slope of HI over 60 min (trend direction)

  Why these 5 derived features?
  - hi_now gives the current state
  - hi_mean gives the overall level
  - hi_std tells if health is stable or fluctuating
  - hi_min captures the worst-case moment (important for safety)
  - hi_slope tells if the machine is declining, stable, or recovering
  Together, they capture the "position" and "velocity" of the health curve.

  Why NOT elapsed time? An earlier version included elapsed_frac (time since
  start / 800 min). It had 17% feature importance but is invalid in production:
  the engine's second_counter resets on server restart and grows unbounded
  past 800 min. The RF never saw values > 1.1 during training, causing
  extrapolation risk. Removing it dropped R² by only 0.2% (0.985 → 0.983),
  confirming the sensor + HI features capture the degradation signal properly.

  TRAINING LABEL (what we're predicting):
    RUL_minutes = t_fail - t_now
  where t_fail is the first time simulated_hi (ground truth) drops below 0.3
  (Critical threshold) AFTER the machine was healthy. IMPORTANT: we use
  simulated_hi (not hi_smooth) for the label because the ground truth gives
  clean, noise-free failure points. The model then learns to predict from
  noisy features (hi_smooth, sensor data) to clean targets. This separation
  of label quality from feature noise was the KEY design decision that boosted
  R² from 0.65 to 0.985.

  The 60-minute lookback (RUL_LOOKBACK_MIN = 60) means the model only starts
  making predictions after it has accumulated 60 minutes of HI history.

  CONFIDENCE INTERVAL:
  Instead of just taking the mean of 200 trees, we compute:
    - RUL_predicted = mean of all 200 tree predictions
    - CI_low        = 10th percentile of tree predictions
    - CI_high       = 90th percentile of tree predictions
  This gives an 80% confidence interval — "we're 80% sure the real RUL falls
  between CI_low and CI_high." Wide CI = high uncertainty (e.g. noisy profile D).
  Narrow CI = high confidence (e.g. linear profile A).

  CONVERSION TO DAYS:
  The model predicts RUL in minutes of operating time. We convert to calendar
  days using: RUL_days = RUL_minutes / RUL_MIN_TO_DAY = RUL_minutes / 9.
  (Each trajectory of 800 min represents 90 calendar days.)

  RESULTS (on test set):
  - R² = 0.983  (target was > 0.85 — far exceeded)
  - RMSE = 25.4 min operating time = 2.82 calendar days
  - MAE  = 12.8 min operating time = 1.43 calendar days
  - Per-profile (all 4 profiles covered):
    A_linear:       RMSE=1.85 days, R²=0.993
    B_exponential:  RMSE=1.22 days, R²=0.997
    C_stepwise:     RMSE=1.96 days, R²=0.992
    D_noisy_linear: RMSE=4.97 days, R²=0.906
  - GBR comparison: R²=0.984 vs RF=0.983 — RF chosen for CI + SHAP
  - Best on B_exponential, worst on D_noisy_linear (expected:
    noisy data makes prediction harder)

  OUTPUT: data/processed/rul_predictions.csv (true vs predicted RUL + CI)
          models/random_forest_rul.pkl (trained RF model, 300 trees)

   STEP 6B — NASA CMAPSS CROSS-VALIDATION (step6b_cmapss.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  "Your simulation is not real data — how do I know your pipeline works on
  actual machines?" This is the jury's most likely question. To answer it, we
  apply the EXACT same pipeline to NASA's CMAPSS benchmark — a public dataset
  of 100 real turbofan engines running to failure. If our scores are
  competitive with published research, the pipeline is validated.

Technical:

  DATASET: NASA CMAPSS FD001
  - 100 turbofan engines, each running until failure
  - 21 sensor channels + 3 operational settings
  - Published by NASA Prognostics Center of Excellence
  - Standard benchmark for Remaining Useful Life prediction

  IDENTICAL PIPELINE APPLIED:
  1. Feature engineering: rolling means (20, 50 cycles), rolling std, EWM,
     first differences for each of 14 useful sensors (7 sensors × 2 features
     + more). Total: ~85 features per timestep.
  2. Z-score normalization on first 20 cycles (= healthy reference)
  3. Isolation Forest trained on healthy data only (cycle ≤ 20)
  4. HI computation from IF anomaly scores (same p5/p95 normalization)
  5. HI smoothing with 20-cycle rolling mean
  6. RUL prediction with RandomForest (500 trees this time — more data available)

  RUL CAPPING: Standard CMAPSS practice caps RUL at 125 cycles. The reasoning:
  early in an engine's life, the health is perfect and predictions of "this will
  last another 290 cycles" vs "310 cycles" are both essentially "it's fine."
  Capping focuses the model on the critical prediction zone (< 125 cycles left).

  NASA SCORING FUNCTION:
  The standard scoring function penalizes LATE predictions more than early ones:
    If predicted < true: penalty = e^(-error/13) - 1  (light penalty for caution)
    If predicted > true: penalty = e^(error/10) - 1   (heavy penalty for over-optimism)
  This makes physical sense: predicting failure too late is dangerous.

  RESULTS:
  - RMSE ≈ 14.1 cycles (surpasses literature targets: 18.4 range)
  - MAE  ≈ 9.6 cycles
  - R²   ≈ 0.89
  These numbers prove the pipeline generalizes beyond the elevator context to
  a completely different domain (turbofan engines) with real failure data.

  OUTPUT: outputs/cmapss_metrics.json + outputs/plots/plot6_cmapss.png


════════════════════════════════════════════════════════════════════════════════
   STEP 7 — EXPORT & PRODUCTION ENGINE (step7_export.py)
════════════════════════════════════════════════════════════════════════════════

Non-technical:
  Everything we've built so far works in "batch mode" — process a whole dataset
  at once. But in production, the elevator sends 1 sensor reading per second
  via MQTT, and we need to respond instantly. Step 7 packages all models into
  a single "engine" class that can process real-time data.

Technical:

  THE PREDITEQ ENGINE (models/prediteq_engine.py):
  A Python class with internal state, designed to be instantiated once per
  machine and kept alive in memory by the FastAPI backend.

  Key methods:
  - update(raw_features) — called every second with new sensor readings.
    1. Validates the payload (all 12 features present?)
    2. Rejects outliers (> 5σ from healthy mean — prevents sensor glitches
       from corrupting the model)
    3. Normalizes using stored scaler_params.json
    4. Computes IF anomaly score
    5. Computes hybrid score (IF + RMS z-score, weighted by alpha=0.6)
    6. Updates rolling buffer (deque, max 600 points)
    7. Computes smoothed HI from buffer
    8. Every 60 seconds, appends HI to the RUL buffer
    Returns: { hi_smooth, zone, score_if, buffer_lengths, uptime }

  - predict_rul() — called every 10 seconds by FastAPI scheduler, but only
    produces a new prediction when the HI buffer has fresh data (every 60 engine-seconds).
    Requires 60 minutes of HI history (warming-up phase until then).
    Builds the 17-feature vector (12 sensor + 5 HI summary).
    Runs all 300 RF trees, computes mean + 10th/90th percentile CI.
    Returns: { rul_days, ci_low, ci_high, status }

  - reset_after_maintenance() — called when a technician marks a maintenance
    task complete. Clears all buffers so the engine starts fresh.

  SAFETY FEATURES:
  - Incomplete payload → logged warning, returns UNKNOWN zone (doesn't crash)
  - Outlier rejection → prevents corrupted readings from affecting HI
  - Adaptive warm-up → uses min_periods=1, so HI is available from second 1
    but becomes more accurate over time
  - Buffer limits → deque(maxlen=600) prevents memory growth

  MQTT SCHEMA:
  Step 7 also exports outputs/mqtt_schema.json — the complete API contract:
  - Input: machine_id, timestamp, rms_mms, power_kw, current_a, pf, temp_c, humidity_rh
  - Output: hi_smooth, zone, rul_days, ci_low, ci_high
  - Alert logic: Degraded → weekly email, Critical → immediate email
  - Data frequency specifications: 1 Hz MQTT, 600-point HI buffer, 60-min RUL buffer

  VALIDATION:
  Step 7 runs a self-test:
  1. Feed 70 seconds of healthy data → verify HI ≈ 1.0, Zone = Excellent
  2. Feed 3600 seconds (1 hour) → verify RUL prediction works
  3. Test outlier rejection (rms = 999.0) → verify safe error return
  4. Test incomplete payload → verify graceful handling
  5. Test maintenance reset → verify clean buffer state

What the ML pipeline produces:
The 7 steps train 2 models (Isolation Forest + Random Forest) and export a PrediteqEngine class. In production, the FastAPI backend instantiates one engine per machine. Each engine receives real-time sensor data via MQTT (1 Hz) and outputs: HI (0-100%), zone (Excellent/Good/Degraded/Critical), RUL (days), and confidence intervals. These values get written to the Supabase machines table, and the frontend reads them.

Current state of the 3 machines:
All 3 machines (ASC-A1, ASC-B2, ASC-C3) exist in the Supabase database and the frontend reads from it via useMachines.ts. The static fallback in machines.ts has hardcoded demo values:

ASC-A1: HI=0.87, RUL=142j, status=ok (healthy machine)
ASC-B2: HI=0.62, RUL=54j, status=degraded (mid-life)
ASC-C3: HI=0.31, RUL=12j, status=critical (near failure)
The simulation covers ALL machines equally. The 100 trajectories in Step 1 don't correspond to one specific machine — they train a generic model that works on any SITI FC100L1-4 elevator. The same trained engine is instantiated 3 times (once per machine) by FastAPI. Each receives its own sensor stream and maintains its own buffers.

For demo purposes: Yes, without a live MQTT sensor feed, the 3 machines show their Supabase-stored values (which the backend simulator updates). The HI chart (genHI() function) generates a synthetic 90-day trend from the current HI value. So all 3 machines are "live" in the demo — they just reflect the last values the backend wrote, not placeholder text.

In short: ML trains 1 generic model → deployed 3 times (1 per machine) → each one processes its own sensor stream independently → writes to Supabase → frontend reads it. All 3 machines are functional, not just 1.
